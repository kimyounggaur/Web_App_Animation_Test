## 📖 0. 이 문서 사용법

이 문서는 **Fable 5에게 그대로 붙여넣는 프롬프트 모음**입니다.

- **방법 A (권장):** `Phase 0` → `Phase 8`을 **순서대로** 하나씩 입력. 각 단계가 누적되며 안정적으로 발전합니다.
- **방법 B (빠른 시작):** 맨 아래 **`§9 단일 메가 프롬프트`** 하나만 입력. Fable 5가 전체를 한 번에 설계·구현합니다.
- `{ }` 안의 값은 본인 상황에 맞게 **교체**하세요.

```

운영하고 있어. URL: https://web-app-animation-test.vercel.app/

현재 구조는 다음과 같아:
- 좌측 사이드바 네비게이션 + 우측 긴 단일 스크롤 페이지(앵커 이동)
- 전역 컨트롤: 테마 토글 / 전역 속도(0x~2x) / 정지 / 모션 줄이기 / 리셋

  · Start: Overview
  · CSS: A.기초 메커니즘 / B.마이크로 인터랙션 / C.UI 컴포넌트 / D.스크롤&전환 / E.미디어 UI

[현재의 문제점 — 내가 느끼는 것]

6. 마음에 든 레시피를 모아두거나 다시 찾을 수 없다

지금부터 이걸 단계적으로 업그레이드할 거야.
아직 코드를 작성하지 말고, 이 컨텍스트를 이해했다고만 답해줘.
```

## 🎯 2. 업그레이드 목표 & 성공 기준 (Fable 5에게 함께 입력)

```
[업그레이드 핵심 목표]
"전문가에게는 빠른 도구, 초보자에게는 친절한 안내서"가 되는 것.

[반드시 지켜야 할 제약 — 절대 위반 금지]
1. 기존 데모/코드/기능을 절대 삭제하지 마라. UI/UX만 개선한다.
2. "Live-first · Honest code" 철학 유지 — 복사 코드는 프레임워크 없이 동작해야 한다.
3. 전역 컨트롤(속도/정지/모션 줄이기)은 모든 신규 UI에도 적용되어야 한다.
4. 접근성은 옵션이 아니라 기본값 — prefers-reduced-motion, 키보드, 포커스 가시성 준수.
5. 외부 무거운 라이브러리 금지(기존 방침 유지). 필요한 건 바닐라로.

[성공 기준 — 이 6가지로 평가한다]
□ 발견성: 어떤 데모든 3번의 클릭/입력 안에 도달 가능
□ 첫인상: 처음 본 사람이 10초 안에 "무엇을 하는 곳인지" 이해
□ 개성: 흔한 템플릿이 아닌, '모션 실험실'만의 시각 정체성
□ 회수성: 마음에 든 항목을 저장하고 다시 찾을 수 있음
□ 반응성: 모바일에서도 데스크톱만큼 쾌적
□ 일관성: 모든 화면이 같은 디자인 토큰/규칙을 따름

이해했으면 "확인"이라고만 답해줘.
```

## 🎨 3. 디자인 방향 확립 (Phase 0의 핵심)

> 이 단계가 전체 품질을 좌우합니다. Fable 5에게 **먼저 디자인 시스템을 설계하고 자기비평**하게 만드세요.

```
[Phase 0 — 디자인 시스템 설계]

너는 지금 "모션을 다루는 정밀 계측 도구"를 디자인하는 스튜디오의 디자인 리드야.
흔한 AI 기본값(크림+세리프+테라코타 / 블랙+형광 / 신문 레이아웃)은 거부당했어.
이 앱만의 정체성이 필요해.

[디자인 테제 — 출발점으로 제안]
"이 랩은 그 자체로 최고의 데모다."
모션을 '측정하고 조율하는 재료'로 다루는 계측 벤치(calibration bench).
- 시그니처 모티프: easing 곡선(cubic-bezier). 핸들을 드래그하는 그 곡선이
  브랜드 마크이자 실제 컨트롤로 반복 등장.
- 컬러: 기존 앱의 'motion spectrum'(인디고 #4f5bd5 → 시안 #1aa6c4 → 그린 #1f9d57)을
  '신호색'으로 승격. 나머지는 계측기급 중립색(잉크/그래파이트/페이퍼)으로 절제.
- 모션 자체는 절제 — 실험실답게 "목적 있는 모션"만. 모든 모션은 전역 속도/정지에 반응.

[너의 작업 — 코드 작성 전에 반드시 이 순서로]
1. 위 테제를 채택하거나, 더 나은 대안이 있으면 제안하고 이유를 설명해라.
2. 디자인 토큰을 확정해라:
   · Color: 4~6개 named hex (배경/표면/텍스트/신호/경계/강조)
   · Type: 디스플레이 폰트 + 본문 폰트 + 유틸(데이터/캡션용 mono) — 3개 역할.
     흔하지 않은 조합으로 골라라. 한국어 가독성도 고려해라.
   · Spacing/Radius/Shadow: 일관된 스케일
   · Motion: easing 토큰, duration 토큰
3. 확정한 토큰을 흔한 기본값과 비교하며 자기비평해라.
   "다른 앱에도 그대로 쓸 수 있는 선택"이면 다시 골라라. 무엇을 왜 바꿨는지 적어라.
4. 최종 토큰을 CSS custom properties(:root) 형태로 출력해라.

아직 페이지 전체를 만들지 마라. 디자인 토큰 시스템과 그 근거만 보여줘.
```

✅ **완료 기준:** 명확한 디자인 테제 + 근거 있는 토큰 시스템(CSS 변수) + 자기비평 기록이 나왔다.

## 🏗️ 4. 단계별 실행 프롬프트

### Phase 1 — 정보 구조(IA) 재설계

```
[Phase 1 — 정보 구조 재설계]

30개 이상 데모를 사람이 길을 잃지 않도록 재구성해줘. 코드 작성 전에 설계부터.

1. 현재 섹션(A~J + Advanced Labs)을 사용자 관점의 상위 카테고리로 재그룹화해라.
   힌트: "무엇을 만들고 싶은가" 기준(예: 피드백 / 로딩 / 전환 / 데이터 / 3D·물리 /
   사운드 / 스토리텔링)으로 재배치하되, 기존 모듈명은 보조 라벨로 유지.
2. 각 데모에 메타데이터 태그 체계를 설계해라:
   · 엔진(CSS / SVG / JS / Canvas / WebGL / Audio)
   · 난이도(입문 / 중급 / 고급)
   · 용도(피드백 / 로딩 / 전환 / 데이터 / 장식 / 내비)
   이 태그는 이후 필터링에 쓰인다.
3. 결과를 트리 구조 + 태그 매핑 표로 보여줘. 아직 UI는 만들지 마라.
```

✅ **완료 기준:** 사용자 중심 카테고리 트리 + 데모별 태그 매핑이 정리됐다.

---

### Phase 2 — 탐색 시스템 (네비게이션 · 검색 · 갤러리)

```
[Phase 2 — 탐색 시스템 구축]

Phase 1의 IA와 Phase 0의 디자인 토큰을 바탕으로 탐색 경험을 만들어줘.
다음을 모두 구현하되, 각각 독립적으로 동작하고 서로 충돌하지 않게 해라.

1. [갤러리/오버뷰 모드]
   - 모든 데모를 카드 그리드로 한눈에 보는 화면을 추가해라.
   - 각 카드: 미니 라이브 프리뷰(또는 정지 시 대표 프레임) + 제목 + 태그 칩.
   - 카드 클릭 → 해당 데모 상세로 부드럽게 이동(View Transition API, 미지원 시 fallback).
   - "갤러리 ↔ 문서(긴 스크롤)" 뷰를 토글할 수 있게 해라.

2. [실시간 검색 + 필터]
   - 상단에 검색 입력. 제목/태그/코드 키워드로 즉시 필터링(디바운스 적용).
   - Phase 1의 태그(엔진/난이도/용도)로 멀티 필터 칩. 복수 선택 가능.
   - 결과 없을 때: "검색 결과 없음"이 아니라 행동을 유도하는 빈 상태
     (예: 인기 데모 추천 + 필터 초기화 버튼).

3. [커맨드 팔레트] (Cmd/Ctrl+K)
   - 키보드만으로 모든 데모/섹션으로 점프. 퍼지 검색.
   - 최근 본 항목, 즐겨찾기(Phase 5)도 노출.
   - 화살표 키 이동, Enter 이동, Esc 닫기 — 완전한 키보드 접근성.

4. [스크롤 진행 + 현재 위치]
   - 긴 문서 뷰에서 현재 섹션을 사이드바에 하이라이트(scroll-spy).
   - 상단에 얇은 읽기 진행률 바.

구현 후, 각 기능이 모바일에서도 동작하는지 스스로 점검하고 문제를 보고해라.
```

✅ **완료 기준:** 갤러리·검색·필터·커맨드 팔레트·scroll-spy가 모두 동작하고 서로 충돌하지 않는다.

### Phase 3 — 첫 방문 경험 (온보딩)

```
[Phase 3 — 첫 방문자 온보딩]

처음 온 사람이 10초 안에 가치를 이해하게 만들어줘.

1. [히어로 재설계]
   - Phase 0의 '시그니처 모티프(easing 곡선)'를 히어로의 주인공으로.
   - 단순 큰 숫자+라벨 같은 템플릿 금지. 이 앱의 정체("보고, 만지고, 복사하는")를
     즉시 보여주는 인터랙티브 순간을 만들어라 — 예: 방문 즉시 살아있는 미니 데모.
   - 핵심 CTA 2개: "둘러보기(갤러리)" / "바로 시작(추천 데모)".

2. [3스텝 가이드 투어] (선택적, 건너뛰기 가능)
   - 최초 방문 시에만 표시(localStorage로 기억).
   - 3단계만: ① 데모를 만진다 ② 컨트롤을 바꾼다 ③ 코드를 복사한다.
   - 각 단계는 실제 UI 요소를 가리키는 가벼운 코치마크. 강제하지 않는다.

3. [추천 시작점]
   - "처음이라면 여기부터" 큐레이션 3~5개를 히어로 아래에 배치.

온보딩은 재방문자에게 방해가 되면 안 된다. 그 점을 어떻게 처리했는지 설명해라.
```

✅ **완료 기준:** 개성 있는 히어로 + 건너뛸 수 있는 투어 + 추천 시작점. 재방문 시 자동 노출 안 됨.

### Phase 4 — 데모 카드 & 코드 경험 개선

```
[Phase 4 — 데모 상세 & 코드 UX]

각 데모 카드와 코드 영역을 더 직관적으로 만들어줘.

1. [데모 카드 표준화]
   - 모든 데모를 일관된 레이아웃으로: [프리뷰 영역] / [컨트롤] / [코드] 명확히 구분.
   - 컨트롤 변경 → 프리뷰 + 코드가 동시에 갱신됨을 시각적으로 분명히(미세한 하이라이트).
   - "다시 재생", "정지", "리셋"을 카드 단위로도 제공.

2. [코드 영역 개선]
   - 구문 강조(가벼운 자체 하이라이팅, 외부 라이브러리 없이).
   - 복사 버튼: 클릭 시 "복사됨" 피드백(시각 + 선택적 사운드, 기존 Web Audio 활용).
   - 코드 탭: 같은 데모를 CSS / JS / HTML 등으로 전환 가능하면 탭 제공.
   - "이 코드 왜 이렇게?" 토글 — 핵심 라인 주석/설명 펼치기.

3. [복사 코드 신뢰성]
   - 복사되는 코드는 빈 HTML 파일에 그대로 붙여도 동작해야 한다(기존 원칙).
   - 의존성이 필요한 경우 주석으로 명시.

기존 데모 중 3개를 샘플로 먼저 적용해 보여주고, 내 확인 후 전체에 일괄 적용해라.
```

✅ **완료 기준:** 표준화된 카드 + 구문 강조 + 신뢰할 수 있는 복사. 샘플 검증 후 일괄 적용.

### Phase 5 — 컬렉션 · 즐겨찾기 · 진행 추적

```
[Phase 5 — 저장과 회수]

마음에 든 항목을 모으고 다시 찾게 해줘. 모두 localStorage 기반(로그인 없음).

1. [즐겨찾기] 각 데모에 별/북마크 토글. 즐겨찾기만 모아보는 필터/탭.
2. [최근 본 항목] 자동 기록, 빠른 복귀.
3. [나만의 컬렉션] 사용자가 데모를 묶어 이름 붙인 모음 생성(예: "내 결제 플로우 모션").
4. [학습 진행 표시] 둘러본 데모 비율을 가볍게 표시(강박적이지 않게, 선택적).
5. [데이터 관리] 즐겨찾기/컬렉션 내보내기(JSON)·가져오기·초기화 제공.

localStorage 용량 한계와 손상 가능성에 대한 예외 처리를 반드시 넣어라.
```

✅ **완료 기준:** 즐겨찾기·최근 항목·컬렉션이 새로고침 후에도 유지되고, 내보내기/초기화가 동작한다.

### Phase 6 — 반응형 & 모바일 우선

```
[Phase 6 — 모바일 경험]

모바일에서 데스크톱만큼 쾌적하게 만들어줘.

1. [네비게이션 변환]
   - 좁은 화면: 사이드바 → 하단 탭 바 또는 슬라이드 드로어로 전환.
   - 커맨드 팔레트는 모바일에서도 접근 가능한 버튼으로 노출.
2. [터치 인터랙션]
   - hover 의존 데모는 터치 대체 동작 제공(tap, long-press).
   - 슬라이더/드래그 컨트롤은 손가락에 맞는 히트 영역(최소 44px).
3. [레이아웃]
   - 카드 그리드는 화면 폭에 따라 1~3열 유동.
   - 코드 영역은 가로 스크롤 또는 줄바꿈 중 가독성 높은 방식.
4. [성능]
   - 저사양 기기 감지 시 동시 실행 애니메이션/입자 수 자동 축소.
   - 보이지 않는 데모는 정지(IntersectionObserver).

대표 브레이크포인트(모바일/태블릿/데스크톱)별로 무엇이 어떻게 바뀌는지 설명해라.
```

✅ **완료 기준:** 모바일에서 네비·터치·레이아웃·성능이 모두 자연스럽다.

---

### Phase 7 — 접근성 & 성능 마감

```
[Phase 7 — 접근성·성능 강화]

기존 J모듈(성능&접근성)과 Motion Hub의 철학을 전체에 일관 적용해줘.

[접근성]
1. 모든 인터랙티브 요소: 키보드 도달 가능 + 보이는 focus-visible.
2. 상태 변화(로딩/성공/오류/토스트)는 aria-live로 스크린리더에 전달.
3. 장식용 SVG는 aria-hidden, 의미 있는 SVG는 title/desc.
4. 색만으로 정보를 전달하지 않음(텍스트/아이콘 병행).
5. prefers-reduced-motion 및 전역 '모션 줄이기'에서 모든 신규 모션이 정지/축소.
6. 명도 대비 WCAG AA 충족.

[성능]
1. transform/opacity 우선, layout 유발 속성 회피(기존 원칙 전수 적용).
2. 화면 밖 애니메이션 정지, 탭 비활성 시 정지.
3. 초기 로딩: 핵심 화면 우선, 무거운 Lab은 지연 로드.
4. FPS 미터를 신규 데모에도 연동 가능하게.

접근성/성능 자가 점검 결과를 체크리스트로 보고해라. 통과 못 한 항목은 솔직히 적어라.
```

✅ **완료 기준:** 키보드/스크린리더/모션 선호/대비/성능 체크리스트를 통과(또는 미통과 항목 명시).

---

### Phase 8 — 마감 디테일 & 통합 QA

```
[Phase 8 — 마감과 QA]

전체를 하나의 완성된 제품으로 다듬어줘.

1. [일관성 감사] 모든 화면이 Phase 0 디자인 토큰을 따르는지 점검.
   어긋난 색/간격/폰트/모션을 찾아 통일해라.
2. [마이크로카피] 버튼/빈 상태/에러/툴팁 문구를 사용자 언어로 통일
   ("Submit"❌ → "저장하기"✅, 동작과 결과의 단어 일치).
3. [빈 상태/에러] 모든 빈 화면을 '행동 유도'로, 에러는 '원인+해결'로.
4. [전환 오케스트레이션] 페이지/뷰 전환이 산만하지 않고 하나의 흐름이 되게.
   (과한 모션은 오히려 AI 티가 난다 — 절제하라.)
5. [최종 자기비평] "악세서리 하나 빼기" 원칙으로 불필요한 장식 1개 이상 제거.
   무엇을 왜 뺐는지 적어라.
6. [회귀 점검] 기존 데모 30개+가 모두 정상 동작하는지 확인.

마지막으로, 변경 요약(무엇을 어떻게 개선했는지)을 사용자 관점에서 정리해줘.
```

✅ **완료 기준:** 디자인 일관성 + 통일된 카피 + 절제된 전환 + 기존 기능 무결성이 확인됐다.

```
[반드시 포함할 업그레이드]

7. 반응형: 모바일 네비(드로어/하단탭), 터치 대체 동작, 유동 그리드, 저사양 자동 축소.

[작업 순서 — 반드시 이대로]
```
